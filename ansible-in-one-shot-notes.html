MODULE 01 Basics & Architecture Beginner

What is Ansible?

Ansible is an open-source, agentless IT automation engine built by Red Hat. It automates configuration management, application deployment, orchestration, and provisioning using simple YAML files called Playbooks. No agent software needs to run on managed nodes — it communicates over SSH (Linux) or WinRM (Windows).

Terraform

  • Infrastructure provisioning
  • Declarative (desired state)
  • Manages cloud resources (EC2, VPC…)
  • Stateful (tracks tfstate)

Ansible

  • Configuration management
  • Procedural + idempotent
  • Configures software on servers
  • Agentless (SSH only)

Architecture

ComponentDescription
Control NodeMachine where Ansible is installed and commands run from
Managed NodesRemote hosts that Ansible configures (no Ansible install needed)
InventoryList of managed nodes (IP/hostnames), static or dynamic
PlaybookYAML file defining ordered list of tasks to run
ModuleUnit of code Ansible sends to managed nodes (e.g. apt, copy)
TaskSingle action calling a module with arguments
RoleReusable, packaged set of tasks, vars, templates
HandlerTask triggered only when notified (e.g. restart nginx)
FactSystem info auto-gathered from managed nodes
VaultEncrypted storage for sensitive variables

Installation

BASHUbuntu/Debian Control Node
# Update packages
sudo apt update && sudo apt upgrade -y

# Install Ansible
sudo apt install ansible -y

# Verify
ansible --version

# Alternative: pip install
pip3 install ansible
💡 Key Concept: Idempotency
Running the same Ansible playbook multiple times produces the same result — it won't reinstall what's already installed or recreate what already exists. This is a core design principle.

The inventory file tells Ansible which hosts to manage and how to connect to them. Default location: /etc/ansible/hosts

MODULE 03 Ad-Hoc Commands Beginner

Ad-hoc commands run a single module on hosts without writing a playbook. Perfect for quick one-off tasks.

BASHSyntax
ansible [pattern] -i [inventory] -m [module] -a "[arguments]" [options]

Common Examples

BASHAd-Hoc Commands
# Ping all hosts
ansible all -m ping

# Run shell command
ansible webservers -m shell -a "uptime"
ansible webservers -m command -a "df -h"

# Install a package (Ubuntu)
ansible webservers -m apt -a "name=nginx state=present" -b

# Install a package (RHEL/Amazon Linux)
ansible dbservers -m yum -a "name=httpd state=present" -b

# Copy a file
ansible all -m copy -a "src=/tmp/app.conf dest=/etc/app.conf"

# Gather facts about hosts
ansible web1 -m setup

# Reboot servers
ansible all -m reboot -b

# Check free memory
ansible all -m shell -a "free -m"
FlagMeaning
-mModule name
-aModule arguments
-iInventory file path
-bBecome (sudo)
-uRemote user
--private-keySSH private key path
-v / -vvvVerbosity (increase for debug)
MODULE 04 Playbooks Beginner

Playbooks are YAML files that define a series of plays. Each play maps a group of hosts to a set of tasks. This is Ansible's core automation mechanism.

Anatomy of a Playbook

YAMLinstall-nginx.yml
---
# A playbook is a list of plays
- name: Install and start Nginx          # Play name
  hosts: webservers                         # Target group/host
  become: true                              # Run as sudo
  gather_facts: true                        # Collect system info

  tasks:                                     # List of tasks

    - name: Update apt cache
      apt:
        update_cache: yes

    - name: Install nginx
      apt:
        name: nginx
        state: present                        # present | absent | latest

    - name: Start and enable nginx service
      service:
        name: nginx
        state: started
        enabled: yes

    - name: Copy config file
      copy:
        src: files/nginx.conf
        dest: /etc/nginx/nginx.conf
        owner: root
        group: root
        mode: '0644'
      notify: Restart Nginx                   # Trigger handler

  handlers:
    - name: Restart Nginx
      service:
        name: nginx
        state: restarted

Running Playbooks

BASH
# Basic run
ansible-playbook -i inventory.ini playbook.yml

# Dry run (check mode — no changes)
ansible-playbook -i inventory.ini playbook.yml --check

# Run with verbose output
ansible-playbook -i inventory.ini playbook.yml -v

# Limit to specific hosts
ansible-playbook -i inventory.ini playbook.yml --limit web1

# Run specific tags
ansible-playbook -i inventory.ini playbook.yml --tags install

# Skip specific tags
ansible-playbook -i inventory.ini playbook.yml --skip-tags debug

MODULE 05 Variables & Facts Intermediate

Defining Variables

YAMLvariables in a playbook
---
- name: Variable demo
  hosts: all
  vars:
    app_port: 8080
    app_user: deploy
    packages:
      - nginx
      - git
      - curl

  tasks:
    - name: Print variable
      debug:
        msg: "App runs on port {{ app_port }} as {{ app_user }}"

    - name: Install packages
      apt:
        name: "{{ item }}"
        state: present
      loop: "{{ packages }}"

Variable Precedence (Low → High)

  1. Command line values (-e)
  2. Role defaults (defaults/main.yml)
  3. Inventory file variables
  4. group_vars/ files
  5. host_vars/ files
  6. Play vars: section
  7. Task vars:
  8. Extra vars (-e "key=val") — Highest priority

group_vars and host_vars

BASHDirectory Structure
inventory/
├── hosts.ini
├── group_vars/
│   ├── all.yml          # Applies to all hosts
│   ├── webservers.yml   # Applies to webservers group
│   └── dbservers.yml    # Applies to dbservers group
└── host_vars/
    ├── web1.yml         # Applies to web1 only
    └── db1.yml          # Applies to db1 only
YAMLgroup_vars/webservers.yml
nginx_port: 80
app_root: /var/www/html
ssl_enabled: true

Facts (Auto-gathered Variables)

When gather_facts: true (default), Ansible auto-collects system info accessible as variables.

YAMLUsing Facts
# View all facts for a host:
# ansible web1 -m setup

# Common facts in playbooks:
- debug:
    msg: "OS: {{ ansible_os_family }}, IP: {{ ansible_default_ipv4.address }}"

# ansible_os_family → "Debian" | "RedHat"
# ansible_distribution → "Ubuntu" | "CentOS" | "Amazon"
# ansible_hostname → hostname
# ansible_memtotal_mb → total RAM
# ansible_processor_vcpus → CPU count
# ansible_default_ipv4.address → primary IP

Passing Variables at Runtime

BASH
# Pass single variable
ansible-playbook playbook.yml -e "app_port=9090"

# Pass multiple variables
ansible-playbook playbook.yml -e "env=production version=2.1"

# Pass from a vars file
ansible-playbook playbook.yml -e "@vars/production.yml"
MODULE 06 Templates (Jinja2) Intermediate

Templates allow you to generate dynamic configuration files using Jinja2 templating engine. Template files use the .j2 extension.

Basic Template Example

JINJA2templates/nginx.conf.j2
server {
    listen {{ nginx_port }};
    server_name {{ ansible_hostname }};
    root {{ app_root }};

    # Generated by Ansible on {{ ansible_date_time.date }}
    # Host: {{ inventory_hostname }}

    location / {
        proxy_pass http://127.0.0.1:{{ app_port }};
    }
}
YAMLUsing template module in playbook
- name: Deploy nginx config from template
  template:
    src: templates/nginx.conf.j2
    dest: /etc/nginx/nginx.conf
    owner: root
    mode: '0644'
  notify: Restart Nginx

Jinja2 Control Structures

JINJA2Conditionals and Loops in Templates
{# Conditional #}
{% if ssl_enabled %}
    listen 443 ssl;
    ssl_certificate /etc/ssl/cert.pem;
{% else %}
    listen 80;
{% endif %}

{# Loop over list #}
{% for server in backend_servers %}
    server {{ server.ip }}:{{ server.port }};
{% endfor %}

{# Filters #}
{{ app_name | upper }}          {# MYAPP #}
{{ packages | join(', ') }}     {# nginx, git, curl #}
{{ version | default('1.0') }}  {# fallback value #}
MODULE 07 Loops & Conditionals Intermediate

Loops

YAMLloop examples
# Loop over a list
- name: Install packages
  apt:
    name: "{{ item }}"
    state: present
  loop:
    - nginx
    - git
    - curl
    - python3

# Loop over list of dicts
- name: Create users
  user:
    name: "{{ item.name }}"
    groups: "{{ item.groups }}"
    state: present
  loop:
    - { name: alice, groups: sudo }
    - { name: bob, groups: www-data }

# with_items (older syntax, still works)
- name: Create directories
  file:
    path: "{{ item }}"
    state: directory
  with_items:
    - /var/app
    - /var/logs
    - /var/config

Conditionals (when)

YAMLwhen conditionals
# OS-specific tasks
- name: Install nginx on Debian
  apt:
    name: nginx
    state: present
  when: ansible_os_family == "Debian"

- name: Install httpd on RHEL
  yum:
    name: httpd
    state: present
  when: ansible_os_family == "RedHat"

# Multiple conditions
- name: Deploy in production only
  copy:
    src: prod.conf
    dest: /etc/app/
  when:
    - env == "production"
    - ansible_memtotal_mb >= 2048

# Check if variable is defined
- name: Use custom port if defined
  debug:
    msg: "Port: {{ custom_port }}"
  when: custom_port is defined

# Register and check result
- name: Check if file exists
  stat:
    path: /etc/app.conf
  register: app_conf

- name: Print if exists
  debug:
    msg: "Config found!"
  when: app_conf.stat.exists
MODULE 08 Roles Intermediate

Roles are the primary mechanism for organizing playbooks. They package tasks, vars, templates, files, and handlers into a reusable, shareable unit.

Role Directory Structure

BASHStandard Role Layout
roles/
└── nginx/
    ├── tasks/
    │   └── main.yml        # Main task list
    ├── handlers/
    │   └── main.yml        # Handlers (restart nginx, etc.)
    ├── templates/
    │   └── nginx.conf.j2   # Jinja2 templates
    ├── files/
    │   └── index.html      # Static files to copy
    ├── vars/
    │   └── main.yml        # Role variables (high priority)
    ├── defaults/
    │   └── main.yml        # Default variables (low priority)
    ├── meta/
    │   └── main.yml        # Role metadata, dependencies
    └── README.md

Creating a Role

BASH
# Auto-generate role skeleton
ansible-galaxy init roles/nginx

# Result: creates all standard dirs automatically

Role Tasks and Defaults

YAMLroles/nginx/defaults/main.yml
nginx_port: 80
nginx_user: www-data
nginx_worker_processes: auto
app_root: /var/www/html
YAMLroles/nginx/tasks/main.yml
---
- name: Install nginx
  apt:
    name: nginx
    state: present
    update_cache: yes
  tags: [install]

- name: Deploy config
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  notify: Restart Nginx
  tags: [config]

- name: Start nginx
  service:
    name: nginx
    state: started
    enabled: yes
  tags: [service]

Using Roles in a Playbook

YAMLsite.yml
---
- name: Configure web servers
  hosts: webservers
  become: true
  roles:
    - common          # Apply common role first
    - nginx           # Then nginx role
    - role: app
      vars:           # Override role defaults
        app_port: 9090

MODULE 09 Ansible Vault Advanced

Ansible Vault encrypts sensitive data (passwords, API keys, secrets) at rest using AES-256. Never store secrets in plaintext in version control.

Creating and Editing Vaults

BASHVault Commands
# Create a new encrypted file
ansible-vault create secrets.yml

# Edit existing encrypted file
ansible-vault edit secrets.yml

# View without editing
ansible-vault view secrets.yml

# Encrypt an existing file
ansible-vault encrypt vars/passwords.yml

# Decrypt a file
ansible-vault decrypt vars/passwords.yml

# Re-key (change vault password)
ansible-vault rekey secrets.yml

# Encrypt a single string value
ansible-vault encrypt_string 'my_secret_pass' --name 'db_password'

Vault File Content

YAMLvault/secrets.yml (decrypted view)
db_password: "SuperSecretPass123!"
api_key: "sk-abc123def456"
aws_secret_key: "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"

Using Vault in Playbooks

YAMLplaybook.yml
---
- name: Deploy database
  hosts: dbservers
  vars_files:
    - vars/vault/secrets.yml    # Vault-encrypted file
    - vars/main.yml             # Regular vars file
  tasks:
    - name: Configure DB password
      shell: "mysqladmin -u root password {{ db_password }}"
BASHRunning Playbooks with Vault
# Prompt for vault password at runtime
ansible-playbook playbook.yml --ask-vault-pass

# Use a password file (DO NOT commit this file)
ansible-playbook playbook.yml --vault-password-file ~/.vault_pass

# Set in ansible.cfg
# vault_password_file = ~/.vault_pass
🔐 Production Rule
Always add .vault_pass and any decrypted secret files to .gitignore. Only commit encrypted vault files to version control.
MODULE 10 Handlers & Tags Intermediate

Handlers

Handlers are tasks that only run when notified by another task. They run once at the end of the play, regardless of how many tasks notify them. Classic use case: restart a service after a config change.

YAMLhandlers example
tasks:
  - name: Copy nginx config
    template:
      src: nginx.conf.j2
      dest: /etc/nginx/nginx.conf
    notify:
      - Reload Nginx
      - Send Slack Notification

  - name: Update SSL cert
    copy:
      src: cert.pem
      dest: /etc/ssl/
    notify: Reload Nginx   # Won't run twice!

handlers:
  - name: Reload Nginx
    service:
      name: nginx
      state: reloaded

  - name: Send Slack Notification
    uri:
      url: "{{ slack_webhook }}"
      method: POST
      body_format: json
      body:
        text: "Nginx reloaded on {{ inventory_hostname }}"

Tags

Tags let you selectively run parts of a playbook without running everything.

YAMLtags usage
tasks:
  - name: Install nginx
    apt:
      name: nginx
    tags: [install, nginx]

  - name: Configure nginx
    template:
      src: nginx.conf.j2
      dest: /etc/nginx/nginx.conf
    tags: [config, nginx]

  - name: Deploy app code
    git:
      repo: https://github.com/org/app.git
      dest: /var/www/html
    tags: [deploy, app]
BASHRunning with tags
# Run only tasks tagged 'config'
ansible-playbook site.yml --tags config

# Run all except 'deploy'
ansible-playbook site.yml --skip-tags deploy

# Run only 'install' and 'config'
ansible-playbook site.yml --tags "install,config"

# List all available tags
ansible-playbook site.yml --list-tags
MODULE 11 Terraform Integration Advanced

Terraform provisions infrastructure; Ansible configures it. Together they form a complete end-to-end automation pipeline. The course uses Terraform to spin up real AWS EC2 instances (Ubuntu, RHEL, Amazon Linux) and Ansible to configure them.

Architecture

TEXTEnd-to-End Pipeline
┌─────────────────────────────────────────────┐
│  Terraform (terraform apply)                │
│  └── Provision AWS EC2 instances            │
│  └── Output: public IPs, key pairs          │
└──────────────────┬──────────────────────────┘
                   │ Outputs IP addresses
                   ▼
┌─────────────────────────────────────────────┐
│  Ansible Inventory (dynamic / static)       │
│  └── Reads IPs from Terraform outputs       │
│  └── Connects via SSH                       │
└──────────────────┬──────────────────────────┘
                   │ Runs playbooks
                   ▼
┌─────────────────────────────────────────────┐
│  Ansible (ansible-playbook)                 │
│  └── Install software                       │
│  └── Configure services                     │
│  └── Deploy application code               │
└─────────────────────────────────────────────┘

Terraform: Provision EC2 Instances

HCLmain.tf
provider "aws" {
  region = "us-east-1"
}

resource "aws_key_pair" "ansible_key" {
  key_name   = "ansible-key"
  public_key = file("~/.ssh/ansible_key.pub")
}

resource "aws_instance" "ubuntu_node" {
  ami           = "ami-0c02fb55956c7d316"  # Ubuntu 22.04
  instance_type = "t2.micro"
  key_name      = aws_key_pair.ansible_key.key_name

  tags = {
    Name = "ansible-ubuntu"
    Role = "webserver"
  }
}

resource "aws_instance" "rhel_node" {
  ami           = "ami-0583d8c7a9c35822c"  # RHEL 9
  instance_type = "t2.micro"
  key_name      = aws_key_pair.ansible_key.key_name

  tags = {
    Name = "ansible-rhel"
    Role = "dbserver"
  }
}

output "ubuntu_public_ip" {
  value = aws_instance.ubuntu_node.public_ip
}

output "rhel_public_ip" {
  value = aws_instance.rhel_node.public_ip
}

Pattern 1: Static Inventory from Terraform Outputs

BASHWorkflow
# 1. Provision with Terraform
terraform init
terraform apply -auto-approve

# 2. Get outputs
terraform output ubuntu_public_ip  # e.g. 54.12.34.56
terraform output rhel_public_ip    # e.g. 54.12.34.57

# 3. Write to inventory file (manually or via script)
# 4. Run Ansible playbook
ansible-playbook -i inventory.ini site.yml

Pattern 2: Terraform Local-Exec Calls Ansible

HCLmain.tf — Invoke Ansible after provisioning
resource "aws_instance" "web" {
  ami           = "ami-0c02fb55956c7d316"
  instance_type = "t2.micro"
  key_name      = aws_key_pair.ansible_key.key_name

  # Wait for SSH to be ready
  provisioner "remote-exec" {
    inline = ["echo 'SSH is ready'"]
    connection {
      host        = self.public_ip
      type        = "ssh"
      user        = "ubuntu"
      private_key = file("~/.ssh/ansible_key")
    }
  }

  # Run Ansible playbook
  provisioner "local-exec" {
    command = <
      ANSIBLE_HOST_KEY_CHECKING=False \
      ansible-playbook -i '${self.public_ip},' \
        -u ubuntu \
        --private-key ~/.ssh/ansible_key \
        site.yml
    EOT
  }
}

Multi-OS Playbook (Ubuntu + RHEL + Amazon Linux)

YAMLsite.yml — Cross-platform playbook
---
- name: Configure all servers
  hosts: all
  become: true

  tasks:
    # Ubuntu / Debian
    - name: Install nginx (Debian)
      apt:
        name: nginx
        state: present
        update_cache: yes
      when: ansible_os_family == "Debian"

    # RHEL / CentOS
    - name: Install nginx (RedHat)
      yum:
        name: nginx
        state: present
      when: ansible_os_family == "RedHat"

    # Amazon Linux 2
    - name: Install nginx (Amazon Linux)
      command: amazon-linux-extras install nginx1 -y
      when: ansible_distribution == "Amazon"

    # Universal — works on all
    - name: Start nginx
      service:
        name: nginx
        state: started
        enabled: yes

Production Checklist

  • Always use Vault for secrets — never plaintext passwords in git
  • Separate inventories for staging and production
  • Use roles for all reusable logic — no monolithic playbooks
  • Pin role versions in requirements.yml
  • Tag everything so you can run partial deployments safely
  • Use --check (dry-run) before applying to production
  • Enable pipelining in ansible.cfg for performance
  • Use dynamic inventory in cloud environments
  • Store state in git — treat playbooks as code (PRs, reviews)
  • Handle errors with ignore_errors, block/rescue/always

Error Handling

YAMLblock/rescue/always pattern
tasks:
  - block:
      - name: Try to deploy app
        command: ./deploy.sh

      - name: Run migrations
        command: ./migrate.sh

    rescue:
      - name: Rollback on failure
        command: ./rollback.sh

      - name: Notify team
        debug:
          msg: "Deployment FAILED on {{ inventory_hostname }}"

    always:
      - name: Cleanup temp files
        file:
          path: /tmp/deploy_artifacts
          state: absent
