MODULE 01 Basics & Architecture Beginner

What is Ansible?

Ansible is an open-source, agentless IT automation engine built by Red Hat. It automates configuration management, application deployment, orchestration, and provisioning using simple YAML files called Playbooks. No agent software needs to run on managed nodes — it communicates over SSH (Linux) or WinRM (Windows).

Terraform

  • Infrastructure provisioning
  • Declarative (desired state)
  • Manages cloud resources (EC2, VPC…)
  • Stateful (tracks tfstate)

Ansible

  • Configuration management
  • Procedural + idempotent
  • Configures software on servers
  • Agentless (SSH only)

Architecture

ComponentDescription
Control NodeMachine where Ansible is installed and commands run from
Managed NodesRemote hosts that Ansible configures (no Ansible install needed)
InventoryList of managed nodes (IP/hostnames), static or dynamic
PlaybookYAML file defining ordered list of tasks to run
ModuleUnit of code Ansible sends to managed nodes (e.g. apt, copy)
TaskSingle action calling a module with arguments
RoleReusable, packaged set of tasks, vars, templates
HandlerTask triggered only when notified (e.g. restart nginx)
FactSystem info auto-gathered from managed nodes
VaultEncrypted storage for sensitive variables

Installation

BASHUbuntu/Debian Control Node
# Update packages
sudo apt update && sudo apt upgrade -y

# Install Ansible
sudo apt install ansible -y

# Verify
ansible --version

# Alternative: pip install
pip3 install ansible
💡 Key Concept: Idempotency
Running the same Ansible playbook multiple times produces the same result — it won't reinstall what's already installed or recreate what already exists. This is a core design principle.

Ad-hoc commands run a single module on hosts without writing a playbook. Perfect for quick one-off tasks.

BASHSyntax

MODULE 04 Playbooks Beginner

A playbook is a YAML file containing one or more plays. Each play maps a set of hosts to an ordered list of tasks. Playbooks are Ansible's primary automation mechanism — repeatable, readable, and idempotent.

Structure of a playbook

A playbook is a list (-) of plays. Each play has three main sections: the header (who, how), tasks (what to do), and optionally handlers (what to do when notified).

YAMLinstall-nginx.yml
---
- name: Install and configure Nginx      # play name — shown in output
  hosts: webservers                       # inventory group or host to target
  become: true                            # run tasks as sudo/root
  gather_facts: true                      # collect system info before tasks

  vars:                                    # play-level variables
    nginx_port: 80

  tasks:

    - name: Update apt cache
      apt:
        update_cache: yes

    - name: Install nginx
      apt:
        name: nginx
        state: present                      # present | absent | latest

    - name: Deploy config from template
      template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/nginx.conf
        mode: '0644'
      notify: Restart Nginx                 # triggers handler only if changed

    - name: Ensure nginx is started and enabled
      service:
        name: nginx
        state: started
        enabled: yes

  handlers:                               # run once at end of play, only if notified
    - name: Restart Nginx
      service:
        name: nginx
        state: restarted

Play-level keywords

These go at the top of each play (same indent level as hosts:).

KeywordWhat it does
nameHuman-readable label shown in output
hostsTarget group, host, or pattern from inventory (all, webservers, web*)
becomeEnable privilege escalation (sudo) for all tasks in the play
become_userUser to escalate to (default: root)
gather_factsCollect system facts before running tasks (default: true). Set false to speed up plays that don't need facts
varsDefine variables scoped to this play
vars_filesLoad variables from external YAML files
rolesApply one or more roles to the play's hosts
serialRun the play on N hosts at a time (rolling update). E.g. serial: 1 or serial: "30%"
max_fail_percentageAbort the play if more than N% of hosts fail
any_errors_fatalStop the entire play immediately if any host fails
environmentSet environment variables for all tasks in the play
tagsTag the entire play so it can be selected or skipped with --tags

Task-level keywords

These go on individual tasks inside tasks:.

KeywordWhat it does
nameLabel for the task shown in output — always write one
whenConditional — task runs only if expression is true
loopIterate the task over a list; current item is {{ item }}
registerSave the task result into a variable for later use
notifyTrigger a handler by name if the task reports changed
tagsAssign tags so the task can be run or skipped selectively

failed_whenCustom failure condition based on task output

Multiple plays in one playbook

A single playbook file can contain multiple plays targeting different host groups, running in order.

YAMLsite.yml
---
# Play 1 — configure database servers
- name: Configure DB servers
  hosts: dbservers
  become: true
  tasks:
    - name: Install MySQL
      apt:
        name: mysql-server
        state: present

# Play 2 — configure web servers (runs after Play 1 finishes)
- name: Configure web servers
  hosts: webservers
  become: true
  tasks:
    - name: Install Nginx
      apt:
        name: nginx
        state: present

register & debug — capturing task output

register saves a task's result into a variable. debug prints it. This is how you inspect output, use it in conditions, or pass it to later tasks.

YAML
- name: Check disk usage
  command: df -h /
  register: disk_result          # save output into disk_result

- name: Print the output
  debug:
    var: disk_result.stdout       # .stdout, .stderr, .rc, .changed

- name: Fail if disk is critically full
  fail:
    msg: "Disk critically full!"
  when: "'100%' in disk_result.stdout"

pre_tasks and post_tasks

pre_tasks run before roles and tasks. post_tasks run after everything else. Useful for health checks, notifications, or cleanup.

YAML
- name: Deploy application
  hosts: webservers
  become: true

  pre_tasks:
    - name: Notify deployment started
      debug:
        msg: "Starting deploy on {{ inventory_hostname }}"

  roles:
    - nginx
    - app

  tasks:
    - name: Run smoke test
      uri:
        url: http://localhost/health
        status_code: 200

  post_tasks:
    - name: Notify deployment complete
      debug:
        msg: "Deploy finished on {{ inventory_hostname }}"

Error handling — block / rescue / always

Group tasks in a block to catch failures with rescue (like try/catch) and run cleanup with always.

YAML
- name: Deploy with rollback on failure
  block:
    - name: Run migrations
      command: ./migrate.sh

    - name: Deploy app
      command: ./deploy.sh

  rescue:                              # runs if any task in block fails
    - name: Rollback
      command: ./rollback.sh

  always:                              # always runs regardless of success/failure
    - name: Clean up temp files
      file:
        path: /tmp/deploy
        state: absent

Running playbooks

BASH
# Basic run
ansible-playbook -i inventory.ini playbook.yml

# Dry run — no changes made
ansible-playbook -i inventory.ini playbook.yml --check

# Dry run + show file diffs
ansible-playbook -i inventory.ini playbook.yml --check --diff

# Verbose output
ansible-playbook -i inventory.ini playbook.yml -v

# Limit to one host
ansible-playbook -i inventory.ini playbook.yml --limit web1

# Run only tasks with a tag
ansible-playbook -i inventory.ini playbook.yml --tags install

# Skip tasks with a tag
ansible-playbook -i inventory.ini playbook.yml --skip-tags debug

# Resume from a specific task
ansible-playbook -i inventory.ini playbook.yml --start-at-task "Deploy config"

💡 command vs shell
Use command when you don't need shell features (safer, no injection risk).
Use shell only when you need pipes (|), redirects (>), or environment variable expansion.
MODULE 05 Variables & Facts Intermediate

Defining Variables

YAMLvariables in a playbook
---
- name: Variable demo
  hosts: all
  vars:
    app_port: 8080
    app_user: deploy
    packages:
      - nginx
      - git
      - curl

  tasks:
    - name: Print variable
      debug:
        msg: "App runs on port {{ app_port }} as {{ app_user }}"

    - name: Install packages
      apt:
        name: "{{ item }}"
        state: present
      loop: "{{ packages }}"

Variable Precedence (Low → High)

  1. Command line values (-e)
  2. Role defaults (defaults/main.yml)
  3. Inventory file variables
  4. group_vars/ files
  5. host_vars/ files
  6. Play vars: section
  7. Task vars:
  8. Extra vars (-e "key=val") — Highest priority

group_vars and host_vars

BASHDirectory Structure
inventory/
├── hosts.ini
├── group_vars/
│   ├── all.yml          # Applies to all hosts
│   ├── webservers.yml   # Applies to webservers group
│   └── dbservers.yml    # Applies to dbservers group
└── host_vars/
    ├── web1.yml         # Applies to web1 only
    └── db1.yml          # Applies to db1 only
YAMLgroup_vars/webservers.yml
nginx_port: 80
app_root: /var/www/html
ssl_enabled: true

Facts (Auto-gathered Variables)

When gather_facts: true (default), Ansible auto-collects system info accessible as variables.

YAMLUsing Facts
# View all facts for a host:
# ansible web1 -m setup

# Common facts in playbooks:
- debug:
    msg: "OS: {{ ansible_os_family }}, IP: {{ ansible_default_ipv4.address }}"

# ansible_os_family → "Debian" | "RedHat"
# ansible_distribution → "Ubuntu" | "CentOS" | "Amazon"
# ansible_hostname → hostname
# ansible_memtotal_mb → total RAM
# ansible_processor_vcpus → CPU count
# ansible_default_ipv4.address → primary IP

Passing Variables at Runtime

BASH
# Pass single variable
ansible-playbook playbook.yml -e "app_port=9090"

# Pass multiple variables
ansible-playbook playbook.yml -e "env=production version=2.1"

# Pass from a vars file
ansible-playbook playbook.yml -e "@vars/production.yml"
MODULE 06 Templates (Jinja2) Intermediate

Templates allow you to generate dynamic configuration files using Jinja2 templating engine. Template files use the .j2 extension.

Basic Template Example

JINJA2templates/nginx.conf.j2
server {
    listen {{ nginx_port }};
    server_name {{ ansible_hostname }};
    root {{ app_root }};

    # Generated by Ansible on {{ ansible_date_time.date }}
    # Host: {{ inventory_hostname }}

    location / {
        proxy_pass http://127.0.0.1:{{ app_port }};
    }
}
YAMLUsing template module in playbook
- name: Deploy nginx config from template
  template:
    src: templates/nginx.conf.j2
    dest: /etc/nginx/nginx.conf
    owner: root
    mode: '0644'
  notify: Restart Nginx

Jinja2 Control Structures

JINJA2Conditionals and Loops in Templates
{# Conditional #}
{% if ssl_enabled %}
    listen 443 ssl;
    ssl_certificate /etc/ssl/cert.pem;
{% else %}
    listen 80;
{% endif %}

{# Loop over list #}
{% for server in backend_servers %}
    server {{ server.ip }}:{{ server.port }};
{% endfor %}

{# Filters #}
{{ app_name | upper }}          {# MYAPP #}
{{ packages | join(', ') }}     {# nginx, git, curl #}
{{ version | default('1.0') }}  {# fallback value #}
MODULE 07 Loops & Conditionals Intermediate

Loops

YAMLloop examples
# Loop over a list
- name: Install packages
  apt:
    name: "{{ item }}"
    state: present
  loop:
    - nginx
    - git
    - curl
    - python3

# Loop over list of dicts
- name: Create users
  user:
    name: "{{ item.name }}"
    groups: "{{ item.groups }}"
    state: present
  loop:
    - { name: alice, groups: sudo }
    - { name: bob, groups: www-data }

# with_items (older syntax, still works)
- name: Create directories
  file:
    path: "{{ item }}"
    state: directory
  with_items:
    - /var/app
    - /var/logs
    - /var/config

Conditionals (when)

YAMLwhen conditionals
# OS-specific tasks
- name: Install nginx on Debian
  apt:
    name: nginx
    state: present
  when: ansible_os_family == "Debian"

- name: Install httpd on RHEL
  yum:
    name: httpd
    state: present
  when: ansible_os_family == "RedHat"

# Multiple conditions
- name: Deploy in production only
  copy:
    src: prod.conf
    dest: /etc/app/
  when:
    - env == "production"
    - ansible_memtotal_mb >= 2048

# Check if variable is defined
- name: Use custom port if defined
  debug:
    msg: "Port: {{ custom_port }}"
  when: custom_port is defined

# Register and check result
- name: Check if file exists
  stat:
    path: /etc/app.conf
  register: app_conf

- name: Print if exists
  debug:
    msg: "Config found!"
  when: app_conf.stat.exists
MODULE 08 Roles Intermediate

Roles are the primary mechanism for organizing playbooks. They package tasks, vars, templates, files, and handlers into a reusable, shareable unit.

Role Directory Structure

BASHStandard Role Layout
roles/
└── nginx/
    ├── tasks/
    │   └── main.yml        # Main task list
    ├── handlers/
    │   └── main.yml        # Handlers (restart nginx, etc.)
    ├── templates/
    │   └── nginx.conf.j2   # Jinja2 templates
    ├── files/
    │   └── index.html      # Static files to copy
    ├── vars/
    │   └── main.yml        # Role variables (high priority)
    ├── defaults/
    │   └── main.yml        # Default variables (low priority)
    ├── meta/
    │   └── main.yml        # Role metadata, dependencies
    └── README.md

Creating a Role

BASH
# Auto-generate role skeleton
ansible-galaxy init roles/nginx

# Result: creates all standard dirs automatically

Role Tasks and Defaults

YAMLroles/nginx/defaults/main.yml
nginx_port: 80
nginx_user: www-data
nginx_worker_processes: auto
app_root: /var/www/html
YAMLroles/nginx/tasks/main.yml
---
- name: Install nginx
  apt:
    name: nginx
    state: present
    update_cache: yes
  tags: [install]

- name: Deploy config
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  notify: Restart Nginx
  tags: [config]

- name: Start nginx
  service:
    name: nginx
    state: started
    enabled: yes
  tags: [service]

Using Roles in a Playbook

YAMLsite.yml
---
- name: Configure web servers
  hosts: webservers
  become: true
  roles:
    - common          # Apply common role first
    - nginx           # Then nginx role
    - role: app
      vars:           # Override role defaults
        app_port: 9090

MODULE 09 Ansible Vault Advanced

Ansible Vault encrypts sensitive data (passwords, API keys, secrets) at rest using AES-256. Never store secrets in plaintext in version control.

Creating and Editing Vaults

BASHVault Commands
# Create a new encrypted file
ansible-vault create secrets.yml

# Edit existing encrypted file
ansible-vault edit secrets.yml

# View without editing
ansible-vault view secrets.yml

# Encrypt an existing file
ansible-vault encrypt vars/passwords.yml

# Decrypt a file
ansible-vault decrypt vars/passwords.yml

# Re-key (change vault password)
ansible-vault rekey secrets.yml

# Encrypt a single string value
ansible-vault encrypt_string 'my_secret_pass' --name 'db_password'

Vault File Content

YAMLvault/secrets.yml (decrypted view)
db_password: "SuperSecretPass123!"
api_key: "sk-abc123def456"
aws_secret_key: "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"

Using Vault in Playbooks

YAMLplaybook.yml
---
- name: Deploy database
  hosts: dbservers
  vars_files:
    - vars/vault/secrets.yml    # Vault-encrypted file
    - vars/main.yml             # Regular vars file
  tasks:
    - name: Configure DB password
      shell: "mysqladmin -u root password {{ db_password }}"
BASHRunning Playbooks with Vault
# Prompt for vault password at runtime
ansible-playbook playbook.yml --ask-vault-pass

# Use a password file (DO NOT commit this file)
ansible-playbook playbook.yml --vault-password-file ~/.vault_pass

# Set in ansible.cfg
# vault_password_file = ~/.vault_pass
🔐 Production Rule
Always add .vault_pass and any decrypted secret files to .gitignore. Only commit encrypted vault files to version control.
MODULE 10 Handlers & Tags Intermediate

Handlers

Handlers are tasks that only run when notified by another task. They run once at the end of the play, regardless of how many tasks notify them. Classic use case: restart a service after a config change.

YAMLhandlers example
tasks:
  - name: Copy nginx config
    template:
      src: nginx.conf.j2
      dest: /etc/nginx/nginx.conf
    notify:
      - Reload Nginx
      - Send Slack Notification

  - name: Update SSL cert
    copy:
      src: cert.pem
      dest: /etc/ssl/
    notify: Reload Nginx   # Won't run twice!

handlers:
  - name: Reload Nginx
    service:
      name: nginx
      state: reloaded

  - name: Send Slack Notification
    uri:
      url: "{{ slack_webhook }}"
      method: POST
      body_format: json
      body:
        text: "Nginx reloaded on {{ inventory_hostname }}"

Tags

Tags let you selectively run parts of a playbook without running everything.

YAMLtags usage
tasks:
  - name: Install nginx
    apt:
      name: nginx
    tags: [install, nginx]

  - name: Configure nginx
    template:
      src: nginx.conf.j2
      dest: /etc/nginx/nginx.conf
    tags: [config, nginx]

  - name: Deploy app code
    git:
      repo: https://github.com/org/app.git
      dest: /var/www/html
    tags: [deploy, app]
BASHRunning with tags
# Run only tasks tagged 'config'
ansible-playbook site.yml --tags config

# Run all except 'deploy'
ansible-playbook site.yml --skip-tags deploy

# Run only 'install' and 'config'
ansible-playbook site.yml --tags "install,config"

# List all available tags
ansible-playbook site.yml --list-tags
MODULE 11 Terraform Integration Advanced

Terraform provisions infrastructure; Ansible configures it. Together they form a complete end-to-end automation pipeline. The course uses Terraform to spin up real AWS EC2 instances (Ubuntu, RHEL, Amazon Linux) and Ansible to configure them.

Architecture

TEXTEnd-to-End Pipeline
┌─────────────────────────────────────────────┐
│  Terraform (terraform apply)                │
│  └── Provision AWS EC2 instances            │
│  └── Output: public IPs, key pairs          │
└──────────────────┬──────────────────────────┘
                   │ Outputs IP addresses
                   ▼
┌─────────────────────────────────────────────┐
│  Ansible Inventory (dynamic / static)       │
│  └── Reads IPs from Terraform outputs       │
│  └── Connects via SSH                       │
└──────────────────┬──────────────────────────┘
                   │ Runs playbooks
                   ▼
┌─────────────────────────────────────────────┐
│  Ansible (ansible-playbook)                 │
│  └── Install software                       │
│  └── Configure services                     │
│  └── Deploy application code               │
└─────────────────────────────────────────────┘

Terraform: Provision EC2 Instances

HCLmain.tf
provider "aws" {
  region = "us-east-1"
}

resource "aws_key_pair" "ansible_key" {
  key_name   = "ansible-key"
  public_key = file("~/.ssh/ansible_key.pub")
}

resource "aws_instance" "ubuntu_node" {
  ami           = "ami-0c02fb55956c7d316"  # Ubuntu 22.04
  instance_type = "t2.micro"
  key_name      = aws_key_pair.ansible_key.key_name

  tags = {
    Name = "ansible-ubuntu"
    Role = "webserver"
  }
}

YOUR NOTES C Commands Reference Beginner

Flags — the letters after -

Common single-letter and long-form flags for ansible and ansible-playbook, each with a detailed example.

-m  —  Module name

Specifies which Ansible module to run on the managed nodes. Used with ad-hoc ansible commands.

BASH
ansible -i hosts.ini web -m ping
ansible -i hosts.ini web -m shell
ansible -i hosts.ini web -m command

-a  —  Module arguments

Passes arguments to the module specified by -m. The string is interpreted by the module.

BASH
ansible -i hosts.ini web -m command -a "uptime"
ansible -i hosts.ini web -m shell -a "df -h"
ansible -i hosts.ini web -m apt -a "name=nginx state=present" -b

-i  —  Inventory

Path to the inventory file or a comma-separated list of hosts. Overrides the inventory setting in ansible.cfg.

BASH
ansible-playbook -i hosts.ini site.yml
ansible-playbook -i inventories/production/hosts.ini site.yml

# Inline single host (note the trailing comma)
ansible-playbook -i "54.12.34.56," site.yml -u ubuntu

-e / --extra-vars  —  Extra variables

Passes variables at runtime. These have the highest precedence and override everything else — inventory vars, group_vars, play vars, and role defaults.

BASH
# Pass key=value pairs
ansible-playbook -i hosts.ini site.yml -e "env=prod version=1.2"

# Pass multiple vars
ansible-playbook -i hosts.ini site.yml -e "env=prod" -e "port=9090"

# Pass from a vars file (note the @ prefix)
ansible-playbook -i hosts.ini site.yml -e "@vars/prod.yml"

-u  —  Remote user

SSH username to connect as. Overrides remote_user in ansible.cfg and ansible_user in inventory.

BASH
ansible -i hosts.ini all -m ping -u ubuntu
ansible -i hosts.ini all -m ping -u ec2-user
ansible-playbook -i hosts.ini site.yml -u deploy

-b / --become  —  Privilege escalation

Runs tasks with sudo (or another become method). Required for tasks that need root — installing packages, writing to system dirs, managing services.

BASH
ansible-playbook -i hosts.ini site.yml -b

# Ad-hoc: install a package as root
ansible -i hosts.ini all -m apt -a "name=nginx state=present" -b

-k / --ask-pass  —  SSH password prompt

Prompts for the SSH login password instead of using a key file. Useful when no key is set up yet.

BASH
ansible -i hosts.ini all -m ping -k

-K / --ask-become-pass  —  Sudo password prompt

Prompts for the privilege escalation (sudo) password on the remote host. Use together with -b.

BASH
ansible-playbook -i hosts.ini site.yml -b -K
⚠ Case matters: -k vs -K
-k (lowercase) = SSH login password   |   -K (uppercase) = sudo/become password

-f  —  Forks (parallelism)

Number of hosts Ansible talks to in parallel. Default is 5. Increase for large inventories to speed up runs.

BASH
ansible -i hosts.ini all -m ping -f 20
ansible-playbook -i hosts.ini site.yml -f 50

-c / --connection  —  Connection type

Sets the transport used to connect to managed nodes. Default is ssh. Use local to run on the control node itself without SSH.

BASH
# Run on localhost without SSH
ansible -i hosts.ini localhost -c local -m command -a "whoami"

# Other types: paramiko, winrm, docker
ansible -i hosts.ini windows_host -c winrm -m win_ping

-t / --tags  —  Run tagged tasks only

Runs only the tasks in a playbook that have the specified tag(s). Useful for partial deployments — e.g. only run the install step, skip config and restart.

BASH
# Run only tasks tagged 'install'
ansible-playbook -i hosts.ini site.yml -t install

# Run multiple tags
ansible-playbook -i hosts.ini site.yml -t "install,config"

# Skip a tag
ansible-playbook -i hosts.ini site.yml --skip-tags deploy

# List all available tags without running
ansible-playbook -i hosts.ini site.yml --list-tags

-l / --limit  —  Limit hosts

Restricts the playbook or ad-hoc command to a subset of the inventory — a single host, a group, or a pattern.

BASH
# Limit to a single host
ansible-playbook -i hosts.ini site.yml --limit web1

# Limit to a group
ansible-playbook -i hosts.ini site.yml --limit webservers

# Limit using a pattern (all hosts starting with web)
ansible-playbook -i hosts.ini site.yml -l "web*"

--check  —  Dry run

Runs the playbook without making any actual changes. Ansible reports what would change. Not all modules support check mode fully.

BASH
ansible-playbook -i hosts.ini site.yml --check

--diff  —  Show file differences

Shows a unified diff for any files that would be changed (e.g. by copy or template tasks). Pair with --check to preview changes safely before applying.

BASH
# Preview what files would change and how
ansible-playbook -i hosts.ini site.yml --check --diff

--start-at-task  —  Resume from a task

Starts the playbook run at a specific task name, skipping everything before it. Useful when a run failed mid-way and you want to resume without re-running completed tasks.

BASH
ansible-playbook -i hosts.ini site.yml --start-at-task "Configure app"
ansible-playbook -i hosts.ini site.yml --start-at-task "Start nginx"
💡 Combining flags — real examples
# Dry-run + diff + verbose + limit to one host
ansible-playbook -i hosts.ini site.yml --check --diff -v -l web1

# Override user + extra var + become
ansible-playbook -i hosts.ini site.yml -u ubuntu -e "env=staging" -b

# Run only deploy tag, 20 forks, prompt for sudo
ansible-playbook -i hosts.ini site.yml -t deploy -f 20 -b -K

# Resume a failed run from a specific task
ansible-playbook -i hosts.ini site.yml --start-at-task "Deploy app code" -l web1

YOUR NOTES D Why / What / When Beginner

What is Ansible?

  • Open-source IT automation, configuration management, and orchestration tool maintained by Red Hat.
  • Agentless architecture: connects over SSH (Linux/Unix) or WinRM (Windows) — no agent on managed nodes.
  • Playbooks are YAML files that declare desired state using tasks and modules. Collections provide reusable modules/roles.

Why use Ansible?

  • Simple and human-readable: YAML playbooks are easy to write and review.
  • Agentless: no persistent agents to manage on target hosts.
  • Idempotent: modules apply changes only when needed — safe for repeated runs.
  • Extensible: built-in modules plus Collections and Galaxy for community content.
  • Wide ecosystem: cloud providers, network devices, containers, and Windows.

When to use Ansible?

  • Provisioning infrastructure and configuring servers after provisioning.
  • Deploying applications where declarative configuration fits.
  • Automating routine tasks across many machines (patching, user management, packages).
  • Orchestrating multi-step workflows (deploy DB → migrate → deploy app → smoke tests).
  • Managing network devices where supported modules exist.

When NOT to use Ansible

⚠ Limitations
  • Real-time, low-latency orchestration — not optimized for low-latency control loops.
  • Very large-scale event-driven automation — consider dedicated event-driven platforms.
  • Where a persistent agent with push/pull is required — use Salt, Chef, or Puppet instead.

Key concepts

ConceptDescription
Control nodeMachine where Ansible runs (workstation, CI, or dedicated host)
Managed nodesTarget hosts defined in inventory
InventoryList of hosts — static file or dynamic from cloud APIs
PlaybookYAML document of plays; a play maps hosts → tasks
ModuleUnit of work: yum, apt, copy, template…
RoleReusable grouping of tasks, defaults, handlers, files, templates
CollectionPackaged set of plugins, modules, roles distributed via Galaxy

PLAYBOOKS P1 Variables in Playbooks Intermediate

Variables let you write one playbook that works across different environments, hosts, and configurations by replacing hardcoded values with named placeholders.

1. Inline vars — defined in the play

The simplest way. Defined under vars: in the play header. Scoped to that play only.

YAMLplaybooks/vars.yml
---
- name: Demo inline vars
  hosts: servers
  become: true

  vars:
    username: priyanshu
    app_port: 8080
    packages:
      - nginx
      - git
      - curl

  tasks:
    - name: Print username and port
      debug:
        msg: "User: {{ username }}, Port: {{ app_port }}"

    - name: Install packages
      apt:
        name: "{{ item }}"
        state: present
      loop: "{{ packages }}"

2. vars_files — load from an external file

Keeps variables out of the playbook. Good for separating config from logic, and for having per-environment files.

YAMLvars/app.yml
username: priyanshu
app_port: 8080
app_dir: /var/www/app
YAMLplaybook using vars_files
---
- name: Load vars from file
  hosts: servers
  become: true
  vars_files:
    - vars/app.yml              # loaded before tasks run

  tasks:
    - name: Create app directory
      file:
        path: "{{ app_dir }}"
        state: directory

3. -e / --extra-vars — passed at runtime

Highest precedence. Overrides everything — inline vars, vars_files, group_vars, host_vars. Useful for one-off overrides without editing any file.

BASH
# Pass a single value
ansible-playbook -i hosts.ini playbook.yml -e "username=deploy"

# Pass multiple values
ansible-playbook -i hosts.ini playbook.yml -e "username=deploy app_port=9090"

# Pass from a file
ansible-playbook -i hosts.ini playbook.yml -e "@vars/prod.yml"

4. set_fact — set a variable dynamically during a task

Creates a variable mid-play from computed values or task output. Available to all subsequent tasks in that play.

YAML
- name: Set a dynamic variable
  set_fact:
    deploy_path: "/var/www/{{ username }}/app"

- name: Use the dynamic variable
  file:
    path: "{{ deploy_path }}"
    state: directory

5. register — capture task output as a variable

Saves the result of any task into a variable. The result object has keys like .stdout, .stderr, .rc (return code), .changed.

YAML
- name: Get current date
  command: date
  register: current_date

- name: Print the date
  debug:
    msg: "Today is {{ current_date.stdout }}"

- name: Check if a file exists
  stat:
    path: /etc/app.conf
  register: conf_file

- name: Only run if file exists
  debug:
    msg: "Config found"
  when: conf_file.stat.exists

6. Variable types

YAML
vars:
  # String
  username: priyanshu

  # Number
  app_port: 8080

  # Boolean
  ssl_enabled: true

  # List
  packages:
    - nginx
    - git
    - curl

  # Dictionary / map
  db:
    host: localhost
    port: 5432
    name: myapp

# Accessing dictionary keys:
# {{ db.host }}  or  {{ db['host'] }}

7. Variable precedence (low → high)

When the same variable is defined in multiple places, Ansible uses this order — higher wins:

  1. Role defaults  defaults/main.yml
  2. Inventory file vars
  3. group_vars/all
  4. group_vars/<group>
  5. host_vars/<host>
  6. Play vars:
  7. vars_files:
  8. Role vars  vars/main.yml
  9. set_fact / register
  10. Extra vars -e  — always wins
💡 Quick rule
-e always wins. set_fact and register win over static definitions. Role defaults/ is intentionally the lowest so users can easily override them.
PLAYBOOKS P2 Reading Playbook Output Beginner

When you run ansible-playbook, the output tells you exactly what happened on each host. Here is a real run broken down line by line.

The command

BASH
ansible-playbook -i ../hosts.ini packages.yml -v
  • -i ../hosts.ini — inventory is one directory up
  • packages.yml — the playbook to run
  • -v — verbose mode, shows full task output instead of just a summary

Line 1 — No config file found

OUTPUT
No config file found; using defaults

Ansible looked for ansible.cfg in the current directory, ~/.ansible.cfg, and /etc/ansible/ansible.cfg and found none. It is using built-in defaults. This is just a notice, not an error. Fix it by creating an ansible.cfg in your project root.

PLAY header

OUTPUT
PLAY [Install Packages] ******

Ansible is starting a play named Install Packages — this is the name: field at the top of your playbook. The asterisks just fill the line width.

TASK [Gathering Facts]

OUTPUT
TASK [Gathering Facts] ******
ok: [worker-node-2]
ok: [worker-node-1]

Before running your tasks, Ansible automatically runs the setup module on every host to collect system information (OS, IP, memory, CPU, etc.) called facts. These are stored as variables you can use in your playbook.

  • ok: — facts were gathered successfully, nothing changed
  • Both nodes are contacted in parallel (default 5 forks)
  • Disable this with gather_facts: false in your play to speed things up when you don't need facts

TASK [Install Package]

OUTPUT
TASK [Install Package] ******
changed: [worker-node-1] => {...}
changed: [worker-node-2] => {...}

changed: means the task actually did something — the package was not installed before, so Ansible installed it. The => {...} is the verbose JSON output showing the full result, only visible because you used -v.

The verbose JSON result — key fields

FieldWhat it means
changedtrue — the task made a change on the host
cache_updatedfalse — apt cache was not refreshed (no update_cache: yes in the task)
cache_update_timeUnix timestamp of the last apt cache update
stdoutFull output from apt — what was downloaded, unpacked, installed
stderrAny warnings printed to stderr. Here: W: --force-yes is deprecated
stdout_linesSame as stdout but split into a list, one entry per line
stderr_linesSame as stderr but split into a list

The warning — --force-yes is deprecated

OUTPUT
W: --force-yes is deprecated, use one of the options starting with --allow instead.

This is a warning from apt itself, not from Ansible. It means the playbook (or an older Ansible version) is passing --force-yes to apt, which is deprecated in newer Ubuntu versions. It does not break anything here but can be silenced by upgrading Ansible or removing force: yes from the apt task if present.

💡 How to read the recap at a glance
A healthy run has failed=0 and unreachable=0.
changed=0 on a second run means the playbook is idempotent — everything was already in the desired state, nothing needed to change.

PLAYBOOKS P3 Setup Module & Facts Beginner

The setup module collects system information from managed hosts called facts. Ansible runs it automatically at the start of every playbook (gather_facts: true). You can also run it manually as an ad-hoc command to inspect a host.

The command

BASH
ansible -i ../hosts.ini servers -m setup
  • ansible — ad-hoc, no playbook needed
  • -i ../hosts.ini — inventory one directory up
  • servers — host group to target
  • -m setup — run the setup module, dumps all facts as JSON
ℹ The output is large
A full setup run returns hundreds of facts per host. Pipe it through grep to find what you need, or use the filter argument to narrow it down.

Filtering with grep

Pipe the output through grep to search for specific facts by name.

Example 1 — grep for os_family

BASHCommand
ansible -i ../hosts.ini servers -m setup | grep "os_family"
OUTPUT
        "ansible_os_family": "Debian",
        "ansible_os_family": "Debian",
  • The result appears twice because there are two hosts in the servers group — worker-node-1 and worker-node-2. Each host returns its own facts.
  • ansible_os_family: "Debian" means both nodes are Debian-family systems. Ubuntu is part of the Debian family.
  • This fact is commonly used in playbooks with when: to run different tasks on Debian vs RedHat systems.
YAMLUsing os_family in a playbook
- name: Install nginx on Debian family
  apt:
    name: nginx
    state: present
  when: ansible_os_family == "Debian"

- name: Install nginx on RedHat family
  yum:
    name: nginx
    state: present
  when: ansible_os_family == "RedHat"

Example 2 — grep for distribution

BASHCommand
ansible -i ../hosts.ini servers -m setup | grep "distribution"
OUTPUT

Each field explained:

FactYour valueWhat it means
ansible_distributionUbuntuThe specific Linux distribution name
ansible_distribution_file_parsedtrueAnsible successfully read and parsed the OS release file
ansible_distribution_file_path/etc/os-releaseThe file Ansible read to determine the distribution — standard on all modern Linux systems
ansible_distribution_file_varietyDebianThe upstream family the distro belongs to. Ubuntu is based on Debian

Again the output repeats twice — once per host. Both nodes are running the same OS version.

Filtering with the filter argument instead of grep

Cleaner than grep — returns only matching facts, still in JSON format, one host at a time.

BASH
# Only distribution facts
ansible -i ../hosts.ini servers -m setup -a "filter=ansible_distribution*"

# Only network facts
ansible -i ../hosts.ini servers -m setup -a "filter=ansible_default_ipv4"

# Only memory facts
ansible -i ../hosts.ini servers -m setup -a "filter=ansible_mem*"

# Only CPU facts
ansible -i ../hosts.ini servers -m setup -a "filter=ansible_processor*"
